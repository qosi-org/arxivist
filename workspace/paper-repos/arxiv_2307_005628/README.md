# DNAGPT — Reproduction

**Paper:** [DNAGPT: A Generalized Pre-trained Tool for Versatile DNA Sequence Analysis Tasks](https://arxiv.org/abs/2307.05628)
**Authors:** Daoan Zhang, Weitong Zhang, Yu Zhao, Jianguo Zhang, Bing He, Chenchen Qin, Jianhua Yao (SUSTech, Tencent AI Lab, CityU HK, U Rochester) — 2023

> *Generated by ArXivist from SIR `arxiv_2307_005628`. Independent reproduction, not the authors' official code.*

## What this paper does (plain English)

DNAGPT is a **generalized GPT** for DNA that handles *both* sequence and numerical
data in one model. Beyond standard next-token prediction, it adds two DNA-specific
pre-training tasks — **GC-content regression** (MSE) and **sequence-order prediction**
(is the sequence reversed? CE) — and a **token language** that encodes sequence,
number, instruction, connection, classification and reserved tokens in one space.
Dual **Sequential + Numerical** embeddings and dual **Classification + Regression**
heads let it do classification (GSR recognition), regression (mRNA abundance), and
generation (artificial human genomes) without task-specific architectures.

Combined pre-training loss: **L = 0.01·MSE + CrossEntropy** (Eq 1).

This repo **re-implements the DNAGPT architecture + token language + loss from
scratch** (the architecture is the paper's contribution) — fully **CPU-testable** —
plus an **official-weights loading path** (`from_pretrained` on the authors' `.pth`)
and a **GSR fine-tuning** harness to reproduce Table S2.

## ⚠️ Weights & hardware

- The architecture, token language, loss, and unit tests run **on CPU** — no GPU, no
  custom kernels. `pytest tests/ -q` verifies the paper's design end-to-end.
- Official weights are **`.pth` on Google Drive / Weiyun** (not HF Hub) and load via
  `DNAGPT.from_pretrained` (`strict=False`). See [data/README_data.md](data/README_data.md).
- GSR fine-tuning of the 0.1B model fits on a single mid-range GPU (or CPU, slowly).

## Quick start

```bash
pip install -e .
pytest tests/ -q                                    # verify architecture (CPU)
python data/download.py --weights dna_gpt0.1b_m     # official weights (Drive IDs baked in)
python data/download.py --gsr human_pas_aataaa      # REAL DeepGSR data (Zenodo, ~255 MB)
python train.py --config configs/config.yaml        # fine-tune on Human PAS(AATAAA)
```
The GSR download fetches the authors' exact DeepGSR release and builds the paper's
6:1.5:2.5 splits (11,302 pos + 11,302 hard-negatives for human PAS — matches Table S5).
Without it, `train.py` runs a clearly-labeled **synthetic** smoke test (not comparable
to the paper); with it you get a genuine accuracy vs Table S2's **91.51%**.

## Verify the architecture (no GPU, no weights)

```bash
pytest tests/ -q
```
Checks: non-overlap k-mer → N/k tokens & the 19,530 k-mer vocab (S1.3); GC content;
the classification template; dual heads & causal attention shapes; joint seq+number
forward; the combined loss `L = 0.01·MSE + CE`; and that the GSR head learns.

## Train / Evaluate

```bash
python train.py --config configs/config.yaml --task human_tis_atg
python train.py --config configs/config.yaml --debug          # fast smoke test
python train.py --config configs/config.yaml --dry-run        # build only
python evaluate.py --config configs/config.yaml --checkpoint checkpoints/best.pt
```

## Fine-tuning recipe (Fig S4)

AdamW · betas (0.9, 0.98) · **lr 3e-5** (= 1/3 of pretrain, 0.1B) · weight_decay 1e-1 ·
cosine-with-warmup (3 warmup epochs) · **10 epochs** · batch 8 · grad-clip 1.0 ·
classification head on the final token.

## Reproducibility notes

- **Architecture from scratch**, module names mirror the paper so the official `.pth`
  can be key-mapped (`from_pretrained`, `strict=False`). Reaching Table S2 exactly
  needs those weights (Google-Drive-hosted); from-scratch fine-tuning scores lower.
- **`# ASSUMED`:** numerical-embedding / regression-head MLP shapes (paper says "MLPs"
  without dims — GSR path doesn't use them); AdamW betas over Fig S3's stray "momentum".
- **Token language:** the paper overloads `A`/`N` as both 1-mers and True/False class
  tokens; we keep them distinct (`<TRUE>`/`<FALSE>`) to avoid an id collision — same role.
- **Single seed** vs. the paper's protocol — expect minor variance.
